# MadDragon Mobile Loop and Layout Design

**Date:** 2026-05-15
**Scope:** Mobile-first game loop, screen layout, and planning improvements
**Decision:** Reframe MadDragon as an attack-defense campaign loop with resource production, storage, and raid risk.

## Goals

MadDragon should become a mobile-first RTS campaign where attack and defense feed each other.

The player attacks enemy fortresses to earn resources, uses those resources to improve their base, collects resources produced by buildings, and prepares for raids that can steal resources if the defense fails. The core decision is not only "which unit wins this battle," but "when should I collect, upgrade, attack, or reinforce?"

## Core Loop

```text
Campaign Hub
-> Attack Prep
-> Fortress Attack
-> Reward Result
-> Base Management
-> Raid Defense or Next Attack
```

Attack is the puzzle side of the game. The player scouts, chooses a route, builds a limited army/spell loadout, and breaks the enemy castle.

Defense is the economic pressure side. The player protects stored and owned resources through walls, towers, garrisoned units, headquarters upgrades, and timely collection.

## Resource Economy

Most buildings produce resources over time. Production should be easy to understand from the base screen:

- Gold Mine: produces gold.
- Training or honor buildings: produce honor or upgrade materials.
- Headquarters: stores generated resources and protects part of the player's economy.

Resources exist in two practical states:

- Stored or uncollected resources: generated by buildings and held by the base.
- Owned resources: already collected into the player's wallet.

Raid failure can steal from both states. This gives defense real weight and makes collection timing meaningful.

Example raid loss model:

```text
Defense success: no loss, defense reward granted
Narrow failure: 30% uncollected resources + 5% owned resources lost
Clear failure: 70% uncollected resources + 15% owned resources lost
Headquarters destroyed: 100% uncollected resources + 20% owned resources lost
```

Headquarters level and defensive upgrades should increase protection rate. The player must always see expected loss before a raid or while managing the base.

Example UI copy:

```text
Expected raid loss: 840 / 1,200 stored gold, 400 / 8,000 owned gold
Protection: 35%
Risk: High
```

## Mobile Layout Principles

The battlefield should own most of the screen. UI should appear as compact top information and bottom sheets that open only when needed.

Avoid showing all unit, spell, building, and upgrade controls at once. Mobile screens need one decision layer at a time.

Use:

- Top bar for critical status only.
- Bottom tabs for mode switching.
- Bottom sheets for purchase, build, upgrade, and detail panels.
- Large touch targets for primary actions.
- One dominant call to action per screen.

## Screen Designs

### Campaign Hub

The campaign hub is the player's default orientation screen.

Top bar:

- Current chapter or stage.
- Gold, honor, stars.
- Stored resource indicator.
- Raid risk indicator.

Main area:

- Stage nodes or stage cards.
- Current next recommended action.

Bottom tabs:

- Attack.
- Base.
- Upgrades.

Primary CTA should be either "Start Attack" or "Collect and Manage Base," depending on risk and progression.

### Attack Prep

Attack prep should focus on loadout, not shopping clutter.

Tabs:

- Units.
- Spells.
- Intel.

Units should use squad slots rather than unlimited purchase buttons. A recommended first version is five deploy slots with adjustable counts or unit type assignment.

Spells should be limited to two or three equipped spells for battle. Extra spells can exist in the economy, but battle quick controls should stay short.

Intel should summarize visible enemy threats, route hints, and recommended counters.

### Fortress Attack

The battle screen should keep the map readable.

Top bar:

- Remaining time.
- Target headquarters or castle HP.
- Current earned gold/honor.

Bottom quick bar:

- Selected unit group.
- Deploy or rally.
- Attack.
- Hold/retreat.
- Spell drawer.

The current seven command buttons should be compressed. Lane selection should come mostly from touching the battlefield, not from many command buttons.

### Base Management

Base management is where production, storage, defense, and upgrades meet.

Top bar:

- Stored resources.
- Owned resources.
- Storage capacity.
- Expected raid loss.

Bottom tabs:

- Collect.
- Build.
- Garrison.
- Upgrade.

Important actions:

- Collect All.
- Repair.
- Upgrade Headquarters.
- Improve Protection.

The player should understand why a defensive choice matters. Raid previews should show enemy pressure such as "left cavalry," "center siege," or "right magic."

### Raid Defense

Raid defense should reuse the battle HUD but with defensive goals.

Top bar:

- Headquarters HP.
- Raid timer.
- Estimated protected resources.

Bottom quick bar:

- Garrison selection.
- Rally.
- Focus fire.
- Defensive spell or panic tool.

Result should clearly explain what was protected and what was lost.

### Result

The result screen should prioritize consequence and next action.

Show:

- Victory/defeat.
- Stars.
- Earned or stolen resources.
- Unlocked upgrades or stages.
- Resource protection result.

Primary CTA:

- Base Management after an attack.
- Repair and Reinforce after a failed raid.
- Next Attack only when risk is low enough or the player deliberately chooses it.

## Planning Improvements

The existing prototype has many good systems, but they are visually compressed into a single test UI. The next design pass should improve hierarchy before adding more systems.

Recommended planning changes:

- Define attack and defense as equal halves of the campaign loop.
- Make headquarters a strategic economic building, not just an HP target.
- Give every raid an expected loss preview.
- Split gold, honor, and stars by purpose:
  - Gold: immediate spending, units, spells, buildings, repair.
  - Honor: long-term unlocks and passive upgrades.
  - Stars: campaign progress and reward gates.
- Convert army preparation from many buy buttons to mobile-friendly squad/loadout slots.
- Limit battle spell controls to equipped spells.
- Add raid forecasts so defense planning has a target.

## Implementation Shape

First implementation pass should preserve the current prototype systems and reorganize the UI around the new flow.

Recommended class boundaries for the next pass:

- `CampaignHubScreen`
- `AttackPrepScreen`
- `BattleHud`
- `BaseManagementScreen`
- `DefenseSetupHud`
- `ResultScreen`
- `ResourceStorageSystem`
- `RaidLossCalculator`

The current `TestBootstrap.cs` can remain the integration harness for one pass, but new UI and economy logic should move into focused classes as soon as the layout is validated.

## Testing Strategy

Use EditMode tests for economy and result rules:

- Resource production accumulation.
- Collection behavior.
- Headquarters storage capacity.
- Raid loss calculation.
- Protection rate modifiers.
- Star/reward calculation.
- Unlock conditions.

Use Unity play verification for layout:

- Mobile aspect ratios.
- Touch target size.
- No important text overlap.
- Battlefield visibility with sheets open and closed.
